🏗️ Architecture Overview

The setup uses a push-based webhook — Teams pushes messages to your bot the instant they're sent, instead of polling on a timer. This is why response times are near-instant.

Teams Message │ ▼ Azure Bot Service ──► validates App ID + Secret │ ▼ Cloudflare Tunnel ──► teams.yourdomain.com (public HTTPS) │ ▼ OpenClaw (localhost) ──► processes message, calls Claude/GPT │ ▼ Response sent back via Bot Service API ──► appears in Teams

No open firewall ports. No polling delay. Works from a laptop, home server, or any machine running OpenClaw.

✅ Prerequisites

  • OpenClaw installed — docs.openclaw.ai
  • Azure account — free tier works
  • Cloudflare account — free tier works; cloudflared CLI installed
  • A domain on Cloudflare (or use the free *.trycloudflare.com tunnel for testing)

Step 1 — Azure Bot Registration

1Create a Bot resource in Azure
  1. Go to portal.azure.com
  2. Search for Azure Bot and click Create
  3. Choose Multi Tenant app type
  4. Let Azure auto-create a new Microsoft App ID (or bring your own)
  5. Complete creation — note down the App ID
2Get your Client Secret
  1. In the Bot resource → Configuration → click "Manage Password" link
  2. Under Certificates & secrets, create a new client secret
  3. Copy the secret value immediately — you won't see it again
3Set the Messaging Endpoint
  1. Back in the Bot resource → Configuration
  2. Set Messaging endpoint to:
    https://teams.yourdomain.com/api/messages
  3. Save
4Enable the Teams Channel
  1. In the Bot resource → Channels
  2. Click Microsoft Teams and enable it
  3. Accept the terms, save

Step 2 — Cloudflare Tunnel

This exposes OpenClaw's local webhook server to the public internet via Cloudflare, with no open ports required.

1Install cloudflared 
# macOS
brew install cloudflare/cloudflare/cloudflared

# Login
cloudflared tunnel login
2Create a named tunnel 
cloudflared tunnel create openclaw-teams

This generates a tunnel ID and credential file at ~/.cloudflared/<UUID>.json

3Configure the tunnel

Create ~/.cloudflared/config.yml:

tunnel: <your-tunnel-UUID>
credentials-file: /Users/yourname/.cloudflared/<UUID>.json

ingress:
  - hostname: teams.yourdomain.com
    service: http://localhost:3978
  - service: http_status:404
Port 3978 is the default OpenClaw/Bot Framework port. Adjust if yours differs.
4Add DNS record in Cloudflare 
cloudflared tunnel route dns openclaw-teams teams.yourdomain.com
5Start the tunnel (or run as a service) 
# One-off
cloudflared tunnel run openclaw-teams

# Install as macOS launchd service (auto-starts on boot)
sudo cloudflared service install

Step 3 — OpenClaw Config

Add the Teams plugin to your OpenClaw config. The config file is typically at ~/.openclaw/config.yml or wherever OpenClaw stores its config on your system.

plugins:
  entries:
    msteams:
      enabled: true
      config:
        appId: "YOUR_AZURE_APP_ID"
        appPassword: "YOUR_CLIENT_SECRET"
        tenantId: "YOUR_TENANT_ID"   # optional for single-tenant
Run openclaw gateway restart after changing config to reload the plugin.
Keep appPassword out of version control. Use environment variables or a secrets manager in production.

Verify the gateway is running and Teams is connected:

openclaw gateway status
openclaw status

Step 4 — Add Bot to Teams Chat

The bot can't join chats on its own — it needs to be manually added by a human (this is a Teams permission model constraint).

1Publish the bot (optional for internal use)

For internal org use you can sideload the bot without App Store submission. In Teams Admin Center → Teams apps → Manage apps → Upload an app.

App manifest minimum:

{
  "manifestVersion": "1.1",
  "id": "YOUR_AZURE_APP_ID",
  "version": "1.0.0",
  "packageName": "com.yourorg.openclaw",
  "developer": { "name": "Your Org", "websiteUrl": "https://yourdomain.com",
                  "privacyUrl": "https://yourdomain.com", "termsOfUseUrl": "https://yourdomain.com" },
  "name": { "short": "Revin", "full": "Revin AI Assistant" },
  "description": { "short": "AI assistant", "full": "OpenClaw-powered AI assistant" },
  "icons": { "color": "color.png", "outline": "outline.png" },
  "accentColor": "#0078D4",
  "bots": [{
    "botId": "YOUR_AZURE_APP_ID",
    "scopes": ["personal", "team", "groupChat"],
    "isNotificationOnly": false
  }],
  "permissions": ["identity", "messageTeamMembers"]
}
2Add to a chat or channel
  1. Open the group chat or channel in Teams
  2. Click the + / "Add people" or "Add tab" button
  3. Search for your bot by name and add it
  4. Send a message — the bot will respond via webhook

Step 5 — Subscription OAuth Tokens (Alternative Mode)

This is an alternative approach that uses your Microsoft 365 subscription OAuth tokens instead of a Bot Framework App ID/Secret. Useful when you want the agent to act as a licensed M365 user rather than a bot identity.

1Register an app in Entra ID (Azure AD)
  1. Go to entra.microsoft.com → App registrations → New registration
  2. Set redirect URI to http://localhost:3000/auth/callback (or wherever OpenClaw handles OAuth)
  3. Under API Permissions, add:
    Chat.ReadWrite, ChatMessage.Send, User.Read
  4. Grant admin consent
2Get your OAuth tokens

OpenClaw will walk you through an OAuth flow to get an access + refresh token tied to your M365 account. These tokens act as you in Teams.

openclaw auth add msteams --oauth
Check openclaw help for the exact auth command syntax on your version.
3Update OpenClaw config to use OAuth mode 
plugins:
  entries:
    msteams:
      enabled: true
      config:
        mode: oauth
        # tokens are stored and managed by OpenClaw auth store
OAuth tokens expire. OpenClaw handles refresh automatically if the refresh token is valid. If the agent stops responding, re-run the auth flow.

💡 Tips & Gotchas

  • Webhook vs Polling: Always use webhooks (this setup) over polling. Polling adds 10-60s of lag and burns API quota.
  • Tunnel must be running before Teams tries to deliver messages. Set cloudflared to start on boot.
  • Port 3978 is the Bot Framework default. OpenClaw may use a different port — check openclaw status.
  • Multi-tenant vs Single-tenant: For internal-only bots, single-tenant is more secure. Multi-tenant is needed if the bot will be used across orgs.
  • The bot can't proactively join chats — but once it's added, it can choose when to respond (or not). That's intentional — makes it a participant, not a surveillance tool.
  • Graph API for proactive messaging: If you need the bot to initiate conversations (not just respond), you'll need Chat.Create permissions and Graph API calls. That's a more advanced setup.
  • Logs: openclaw gateway status and the OpenClaw log files are your friend for debugging dropped webhooks.
