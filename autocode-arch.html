Autonomous Software Engineering — Architecture & Usage Guide

Last updated: March 2026 · Maintained by Revin (AI Engineering Assistant)

What Is Code Factory?

Code Factory is an autonomous software engineering system that takes a goal (defined in a program.md file) and iteratively improves a codebase until that goal is met — without human intervention in the loop.

It is not AI-assisted coding (Cursor, Copilot, Claude Code in chat). It is AI-owned coding: the agent reads the spec, breaks down work, writes code, runs tests, scores the result, and decides whether to keep or revert each change — autonomously.

Core Principle

Humans are the interrupt. The agent runs until blocked, then pings a human. Everything else — writing, testing, scoring, reverting — happens without you.

Components

FileRole
program.mdThe spec. Defines goal, success criteria (T1/T2), test commands, scoring target, constraints. Lives in the target repo root.
validate-program.jsPre-flight check. Validates program.md structure before any agent work begins. Exits with code 3 (bootstrap) if baseline is missing.
scorer.jsMeasures code quality via three signals: mutation testing, behavior delta (test pass rate), and complexity. Outputs a single code_bpb score.
loop.jsRuns ONE iteration: invokes agent, scores before/after, keeps or reverts the change, updates session state.
agent.jsRoutes each task to the right model (local Qwen for small changes, Claude Code for architecture-level). Builds the prompt from program.md + history.
supervisor.jsRevin's control layer. Monitors health, adjusts check-in cadence, detects dead sessions, sends steering messages, proposes program.md amendments.

The Scoring Formula

code_bpb = (mutation × 0.50) + (behavior × 0.40) + (complexity × 0.10)
SignalWeightWhat it measuresHow
Mutation50%Does the test suite catch real bugs?Stryker (JS) or mutmut (Python) — kills mutants
Behavior40%Did the change break anything?Test suite pass rate vs locked baseline
Complexity10%Did the code get simpler or more complex?python3 -m lizard cyclomatic complexity delta

Score range: 0.0 (terrible) → 1.0 (perfect). A keep requires the score to improve or hold. A regression triggers an automatic revert.

Loop Exit Codes

CodeMeaningAction
0 — keepIteration improved the scoreCommit the change, run next iteration
1 — discardNo improvement but no regressionRevert, try different approach
2 — revertBehavior regression (tests broke)Hard revert, log failure reason
3 — haltToo many consecutive reverts OR budget exceededStop loop, alert human
5 — doneAll T1 criteria met + target score reached🎉 Create PR, notify human

program.md — The Spec Format

Every Code Factory session needs a program.md in the repo root. Here's the required structure:

# Program: <short title>

## Goal
One paragraph describing what needs to be built or fixed.

## Background / Domain Context
Explain the codebase, relevant files, domain terms, constraints the agent
needs to understand before touching anything.

## Scoring
code_bpb = (mutation × 0.50) + (behavior × 0.40) + (complexity × 0.10)
target_code_bpb: 0.55
weights: {mutation:0.50, behavior:0.40, complexity:0.10}

## Constraints
API budget: $10.00
Max diff per iteration: 300 lines
Language: Python, TypeScript
Protected test files: packages/backend/tests/**, src/**/*.test.*

- Do NOT rename existing public APIs
- Do NOT add new endpoints without flagging for human review

## Success Criteria
# T1 = required, must pass before merge
# T2 = stretch goals, optional
- [T1] The thing that absolutely must happen
- [T1] All existing tests still pass
- [T2] Nice-to-have improvement

## Atomic Decomposition
### Phase 1 — Backend
**Files:** src/app/api/v1/foo.py

1. Do X
2. Do Y

### Phase 2 — Frontend
...

## Test Commands
```bash
cd packages/backend && uv run pytest tests/ -v
```

## Impossibility Constraints
- If X is zero, do not divide — return None
- Never expose field Y to unauthenticated callers
✅ Tip: Run validate-program.js first

Always validate before starting a session. It catches missing labels, undefined targets, missing protected file declarations, and broken test suites before the agent wastes API budget on a broken baseline.

End-to-End Workflow

Step 1 — Write program.md

Write a program.md in the target repo root. Use the Socratic process: interrogate requirements, define T1/T2 criteria explicitly, identify protected files and impossibility constraints before writing a line of code.

Step 2 — Validate

node /path/to/autocode/validate-program.js --program program.md --repo .

Must exit with 0 errors before proceeding. Bootstrap mode (exit 3) is expected on first run — it means baseline hasn't been captured yet.

Step 3 — Capture Baseline

node /path/to/autocode/scorer.js capture-baseline \
  --repo /path/to/repo \
  --test-command "cd packages/backend && uv run pytest tests/ -v" \
  --output /path/to/autocode/sessions/<session-id>/behavior-baseline.json

This locks in the current test pass rate. Any future change that breaks below this gets auto-reverted.

⚠️ Pre-existing failures

If the test suite already has failures, scope the test command to only the passing tests. Code Factory scores relative to baseline — it can't fix what was already broken. A separate session should address pre-existing failures first.

Step 4 — Initialize Supervisor

node /path/to/autocode/supervisor.js start \
  --program /path/to/repo/program.md \
  --repo /path/to/repo \
  --session <session-id>

This initializes the session state and sets the first check-in time (default: 30 minutes).

Step 5 — Run the Loop

cd /path/to/repo && node /path/to/autocode/loop.js \
  --program program.md \
  --repo . \
  --session <session-id> \
  --agent \
  --baseline /path/to/sessions/<session-id>/behavior-baseline.json \
  --results /path/to/sessions/<session-id>/results.tsv \
  --effort medium

Each invocation is one iteration. The loop scores before/after, keeps or reverts, and exits with a code indicating what happened. To run continuously, call it in a shell loop or via supervisor.

Step 6 — Monitor

# Check session status
node supervisor.js status --session <session-id>

# View results log
cat sessions/<session-id>/results.tsv

# Send steering message
node supervisor.js steer --session <session-id> --message "Focus on the forecast.py file first"

Step 7 — PR on Completion

When the loop exits with code 5 (done), all T1 criteria are met. At that point:

  1. Review the kept-attempts diffs in sessions/<id>/kept-attempts/
  2. Push the branch and open a PR
  3. Code Factory's changes are committed on the feature branch — reviewer sees clean incremental commits

Model Routing

The agent routes each task to the cheapest model that can handle it:

ConditionModelWhy
≤ 3 files, no cross-file reasoningLocal Qwen 2.5-Coder-7BFast, free, good for focused edits
> 3 files OR architecture-level changesClaude Code (Sonnet)Stronger reasoning for complex refactors

Re-anchoring happens every 10 iterations — the full program.md + last 5 results rows are injected to reset the agent's working memory and prevent drift.

Session Directory Structure

projects/autocode/
  sessions/<session-id>/
    behavior-baseline.json   # locked test baseline
    loop-state.json          # current iteration state
    checkin-state.json       # supervisor check-in schedule
    results.tsv              # per-iteration scores
    health.json              # model server health
    kept-attempts/           # diffs that improved score
    failed-attempts/         # diffs that were reverted + reason
    pending-amendments.md    # proposed program.md changes
    steering.md              # steering messages from human
  history/
    <repo-slug>/
      history.tsv            # warm-start log across sessions

Supervisor Check-in Cadence

The supervisor adapts its check-in frequency based on session health:

ConditionInterval change
Healthy + progressing× 1.25 (up to 90 min max)
Stuck (score not improving)× 0.75 (down to 10 min min)
Score drift > 2× expectedCheck immediately
No results.tsv update for 30 minSession declared dead → alert

Key Design Decisions

  • Loop is stateless per invocation — each call to loop.js reads state from disk and writes back. No long-running process to babysit.
  • Hard behavior gate — any change that breaks the baseline test suite is auto-reverted, no exceptions. The agent cannot ship a regression.
  • Protected test files — the agent cannot modify test files declared as protected. This prevents it from "gaming" the score by weakening tests.
  • Warm-start compression — history across sessions is compressed to ≤500 tokens so the agent learns from past failures without burning context.
  • Budget enforcement — each session has a dollar cap. Loop halts when exceeded, preventing runaway costs.
